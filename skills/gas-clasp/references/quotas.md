# GAS クォータ・デバッグ・デプロイチェックリスト

応用5 で使う範囲のクォータ・制限と、トラブル時のデバッグ手順、本番稼働前のチェックリストをまとめる。

## 実行時間・サイズ系クォータ

| 項目 | 無料 (Gmail) | Workspace |
|------|-------------|-----------|
| **スクリプト 1 回の実行時間** | 6 分 | 6 分 |
| **カスタム関数 (`=MY_FN()`)** | 30 秒 | 30 秒 |
| **同時実行数** | 30 | 30 |
| **1 日のトリガー実行合計時間** | 90 分 | 6 時間 |
| **トリガー数** | 20 / ユーザー | 20 / ユーザー |

> 💡 月次で数万件のレコード処理が必要なら、**6 分制限に当たる前提で設計** する。`PropertiesService` に「どこまで処理したか」を保存し、次のトリガー発火で続きから再開する形が定石。

## デバッグ手順（うまく動かないとき）

### 1. 実行ログ（Stackdriver）を見る

GAS エディタ > 左メニュー「実行数」（時計マーク） から実行履歴を確認。失敗した実行をクリックすると、`Logger.log()` の出力とエラースタックトレースが見られる。

```javascript
// コード側
function main() {
  Logger.log('main 開始');
  try {
    // ...
    Logger.log('main 完了');
  } catch (e) {
    Logger.log('エラー: ' + e.message + '\n' + e.stack);
    throw e;
  }
}
```

`clasp tail-logs --watch --simplified` でローカルからリアルタイム監視も可能。

### 2. 認可エラー

「承認が必要です」「Authorization required」が出たら:

1. GAS エディタで関数を **手動実行**（▶ボタン）
2. ポップアップで「権限を確認」をクリック
3. 「詳細 > プロジェクト名（安全ではないページ）に移動 > 許可」

スコープが追加されると毎回これが必要。

### 3. 個別の関数を試す

```javascript
function debug() {
  const config = getConfig();
  Logger.log(JSON.stringify(config, null, 2));

  const records = fetchKintoneRecords(config);
  Logger.log(`取得: ${records.length} 件`);
  Logger.log(JSON.stringify(records[0], null, 2));
}
```

GAS エディタで関数選択ドロップダウンから `debug` を選んで実行 → ログ確認。

### 4. トリガーが動かない

| 確認項目 | 確認方法 |
|---------|---------|
| トリガーが登録されているか | GAS エディタ > 左 ⏰「トリガー」 |
| 関数名は正しいか | トリガー設定の「実行する関数」を確認 |
| クォータ超過していないか | 「実行数」で失敗ログを確認 |
| 認可スコープは最新か | 手動実行で再承認 |
| 関数名末尾に `_` が付いていないか | Critical Rules セクション参照 |

### 5. clasp push が通らない

```bash
# 認証アカウント確認
clasp list-scripts

# 違うアカウントだったら
clasp logout
clasp login

# 複数アカウント運用なら
clasp -u personal push
```

## 本番稼働前チェックリスト

応用5 のハンズオン完走後、社内本番運用に上げる前に確認するリスト。

### セキュリティ

- [ ] **API トークン / 秘密情報がコード本文に書かれていない**（grep で `KINTONE_TOKEN` 等を検索して値が出ないか確認）
- [ ] `.gitignore` に `.clasp.json`（scriptId が含まれる）が含まれているか検討
- [ ] **権限スコープが最小** （Kintone トークンは「閲覧」のみ、書き込み権を渡していない）
- [ ] Slack Webhook URL も Properties に保存（コード直書きしていない）
- [ ] GAS プロジェクトの **オーナーが個人アカウントのままになっていないか**（属人化リスク。引継ぎ用の共有ドライブ配下にあるべき）

### 動作・運用

- [ ] **手動実行で main() が成功** することを確認
- [ ] **トリガーが登録されている** ことを確認（GAS エディタ > ⏰ トリガー）
- [ ] **失敗時に Slack 通知** が来ることを確認（一度トークンをわざと壊して試す）
- [ ] スプレッドシートの **共有設定が想定通り**（誰が閲覧 / 編集できるか）
- [ ] **タイムゾーンが Asia/Tokyo** になっているか（`appsscript.json` で確認）
- [ ] **クォータに対する余裕** があるか（1 日のリクエスト数を試算）

### 引継ぎ・属人化対策

- [ ] **README** が書かれている（何をするスクリプトか、誰がオーナーか、止めたいときどうするか）
- [ ] **GitHub に push** されている（属人化防止）
- [ ] **トークンローテーション手順** がドキュメント化されている
- [ ] **IT 部門の承認** が取れているか（社外 API への定期通信を許可済みか）
- [ ] エラーが続いた場合の **エスカレーション先** が決まっている
